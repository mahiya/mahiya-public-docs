# transformers-to-mlx Skill: コードエージェント時代の MLX 移植自動化

## 発行元
- **発行組織**: Hugging Face (MLX Community 協働)
- **著者**: Pedro Cuenca, Awni Hannun, MLX Community
- **発行日**: 2026-04-16
- **URL**: https://huggingface.co/blog/transformers-to-mlx

## 一言概要
Transformers から mlx-lm へのモデル移植を自動化する 15,000 語規模の Skill とテストハーネスを公開し、コードエージェント時代に「人間が開けたであろう PR」を再現可能な形で生成するにゃ。

## ブログで説明している内容

- **背景: コードエージェント時代の OSS 危機**
  - 2026 年、Claude などのコード生成エージェントの普及で Transformers リポジトリは PR ボリュームが月間 10 倍に
  - エージェントは暗黙的設計契約を理解せず低品質 PR を量産
  - Transformers の哲学 (人間による読解優先、平坦な階層、明示的実装) と暗黙的規約をエージェントが破壊
- **MLX と Transformers の関係**
  - MLX は Apple 製オープンソース機械学習フレームワーク
  - mlx-lm は言語モデル実装集
  - Transformers が「真実のソース」、MLX 実装はそこから移植
- **Skill の中身 (約 15,000 語)**
  - 環境セットアップ: 仮想環境構築、Hub からモデル検出/ダウンロード、mlx-lm と Transformers の編集可能インストール
  - アーキテクチャ分析: RoPE 設定検証、safetensors メタデータからの dtype 推論、複数モデル変種間の config 差分分析
  - 実装生成: Transformers コードを参照して mlx-lm 規約に従う自己完結型ファイルを生成
  - 検証テスト: レイヤーごとの数値比較 (Transformers vs MLX)、発散検出とデバッグ、合格まで反復
- **コントリビューター向け機能**
  - スキャフォルディング自動化、RoPE バグ検出 (長シーケンス劣化)、float32 精度汚染検出、分散推論対応
- **レビュアー向けに PR に同梱されるメタデータ**
  - サマリーレポート (変種説明・アーキテクチャ差分)
  - 生成例
  - 数値比較 (相対差異)
  - dtype 検証
  - レイヤーごとの発散ポイント
- **テストハーネス**
  - 目的: LLM 幻覚防止、再現性、透明性
  - 定量チェック (出力 dtype) と定性チェック (長シーケンス挙動、logits 差異 4% は許容か)
  - 出力: JSON 入出力、Markdown サマリー、テストスクリプトのアーカイブ
  - 例: `results/pr-5/2026-04-14T122120-7ce7a68/`
- **インストール**
  - `uv run https://raw.githubusercontent.com/huggingface/transformers-to-mlx/main/install_skill.py`
  - `uvx hf skills add --claude`
  - 例プロンプト: "convert the olmo_hybrid architecture to MLX"
- **対象ユーザーと注意事項**
  - すでに mlx-lm に PR を開いている/開く予定のコントリビューター向け
  - エージェント出力をそのままレビュアーコメントに返さないこと
  - 個人の責任で実装を所有する
- **既知の制限**
  - mlx-vlm 非対応 (Prince Camuma と協力予定)
  - 共有ユーティリティの再構成、量子化モデルアップロード、シンキング構造テストは未対応
- **リソース**
  - https://github.com/huggingface/transformers-to-mlx
  - https://github.com/pcuenca/mlx-lm-tests
  - 例 PR: https://github.com/pcuenca/mlx-lm/pull/5

## 注目ポイントの解説

このブログの核心は「コードエージェント時代の OSS は何で品質を担保するか」という問いへの実用的な回答にゃ。Skill (= 15,000 語のガイダンスドキュメント) で哲学と暗黙規約を明示し、テストハーネスで数値比較を自動化することで、LLM 幻覚を「実装が動くか」というレイヤーで弾く仕組みを作っている。これは規約だけでも自動化だけでもなく、両方を組み合わせて「人間レビュアーが見るべきものに集中できる」状態を作ったところが偉いにゃん。

RoPE バグ検出や float32 精度汚染検出のように、移植時に頻発する具体的な失敗モードを Skill が知識として持っている点も重要にゃ。長シーケンスで段階的に劣化する RoPE 実装は人間でもしばしば見落とす罠で、エージェントが書いたコードでも同じ罠に落ちる。テストハーネスでレイヤーごとの数値比較を強制することで、こうした「動くがじわじわ間違う」バグを発見可能にしているにゃん。

「対象ユーザーは PR レビュー循環に参加意思のあるコントリビューター」と明示している点も、現代のエージェント運用倫理として注目に値するにゃん。エージェント出力をそのまま投げるのではなく、人間が実装の所有者として責任を取ること。これは OSS の文化を守るために Skill 側があえて引いた線で、AI ツールが「自動 PR ファクトリー」化することの弊害を理解した上での設計判断にゃん。

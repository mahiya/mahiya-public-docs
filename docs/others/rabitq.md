# RaBitQ

## 概要

RaBitQ（Randomized Bit Quantization）は、2024年にシンガポール南洋理工大学（NTU）のVectorDBグループによって発表された、高次元ベクトルの近似最近傍探索（ANN）のためのベクトル量子化手法である。SIGMOD 2024で採択された論文で初めて提案され、各ベクトルの次元あたり1ビットという極端な圧縮を実現しながら、理論的に裏付けられた誤差限界を持つ点が最大の特徴である。従来広く使われてきたプロダクト量子化（PQ）やスカラー量子化（SQ）と異なり、RaBitQはコードブックの学習が不要で、ランダム直交回転を用いてベクトルを超立方体の頂点にマッピングする。これにより、32倍の圧縮率を達成しつつ、高次元空間の「測度の集中（Concentration of Measure）」現象を活用して高い検索精度を維持する。2024年9月にはマルチビット対応の拡張版（Extended RaBitQ）が発表され、2ビット～任意のビット幅でのスカラー量子化を最適化する手法に発展した。さらに2026年にはGPUネイティブ実装のIVF-RaBitQがNVIDIA cuVSライブラリに統合された。業界での採用も急速に進んでおり、Milvus、Faiss、Elasticsearch（BBQとして統合）、LanceDB、VectorChord、CockroachDB、Luceneなど主要なベクトルデータベースやライブラリに実装されている。

## 技術者向けポイント

### 1. ランダム直交回転による1ビット量子化の仕組み

- データベクトルをデータセットのセントロイドを基準にセンタリングし、単位ベクトルに正規化した上で、ランダム直交行列Pを適用して回転させる
- 回転後の各次元の符号（正/負）を1ビットとして記録し、超立方体の頂点（各成分が±1/√D）にマッピングする。これにより次元数Dビットのバイナリ文字列でベクトルを表現する
- 距離推定には不偏推定量を用い、量子化ベクトルとクエリの内積を元のベクトルとの内積で補正する。補正係数としてセントロイドまでの距離と量子化ベクトルとの内積の2つのfloat32値を追加保存する
- 理論的にO(1/√D)の誤差限界が証明されており、次元数が高いほど推定精度が向上する。PQのような経験的手法と異なり、漸近的に最適な誤差限界を持つ初めての実用的アルゴリズムである

### 2. Extended RaBitQによるマルチビット量子化への拡張

- 1ビット版ではコードブックが単位球面上の超立方体頂点で構成されるが、マルチビット（2ビット以上）ではグリッド上のコードブックとなり単位球面上にないため、正規化して球面に射影する手法を導入した
- 正規化後のコードブックでは単純な丸め（スカラー量子化）で最近傍ベクトルが見つからないため、データベクトルをリスケーリングしてから丸めることで最適なコードブックベクトルを確実に見つけるアルゴリズムを開発した
- 2ビット量子化では従来のスカラー量子化の最良変種と比較して桁違いの精度向上を達成し、3ビット以上でも一貫した改善を示す。4-bit、5-bit、7-bitで90%、95%、99%のリコールをリランキングなしで達成可能である
- クエリ時の計算は従来のスカラー量子化と全く同じであるため、既存システムへのドロップイン置き換えが可能で、追加のランタイムコストがない

### 3. IVF・HNSW・QGとの統合とGPUネイティブ実装

- RaBitQ Libraryでは、IVF（転置ファイル索引）、HNSW（階層的ナビゲーション可能小世界グラフ）、QG（量子化グラフ）との組み合わせを提供し、メモリ・速度・精度のトレードオフを選択できる
- IVF+RaBitQではクラスタセントロイドを基準にローカルな正規化を行うことでエンコード精度が向上し、量子化コードのみでクエリ処理が完結するため生データへのアクセスが不要になる
- GPU版IVF-RaBitQはCUDAカーネルで全処理をGPU上で完結させ、recall≈0.95でCAGRAの2.2倍のQPS、インデックス構築はCAGRAの7.7倍高速、IVF-PQの4.4倍高速を達成した
- CPU実装ではAVX-512のVPOPCNTDQ命令を活用したビット演算の高速化や、クエリベクトルのSQ4～SQ8量子化による追加の計算削減が可能で、MilvusのIVF_RABITQでは基線IVF_FLATの3倍以上のQPSを達成した

### 4. 実装時の実用的な設計判断と最適化

- Milvus（Knowhere）ではベクトルごとの補助データ3値を全てインデックス構築時に事前計算して保存し、クエリ速度を優先する設計を採用。一方FAISSの実装では1値をクエリ時に計算してメモリを節約する異なるトレードオフを選択した
- Elasticsearchでは実用上リランキングがランダム直交行列の効果を代替できるため、ランダム回転を省略してアルゴリズムとパフォーマンスの複雑さを削減し「BBQ」として統合した
- リランキング戦略はリコール回復に不可欠で、MilvusではSQ8リファインメント付きIVF_RABITQが94.7%のリコールと864QPSを達成。LanceDBでも常にフル精度ベクトルでのリランキングを推奨している
- コードブック学習が不要なためインデックス構築が高速で、データ分布の変化時の再学習も不要。LanceDBのGIST1Mベンチマークではインデックス構築がIVF_PQの130秒に対しRaBitQは21秒と6倍以上高速だった

## ビジネス側向けポイント

### 1. インフラコストの大幅削減

- float32ベクトルを1ビット/次元に圧縮することで最大32倍のメモリ削減を実現する。例えば768次元のベクトル10億件をFP32で保持すると3TB以上のメモリが必要だが、RaBitQでは数十分の一に縮小できる
- MilvusのベンチマークではIVF_RABITQがIVF_FLATの3倍以上のQPSを達成しており、同じハードウェアで3倍のトラフィックを処理可能になる。これはクラウドインスタンスの台数削減に直結する
- コードブックの学習が不要なためインデックス構築が高速で、データ更新時の再学習コストもゼロ。LanceDBのテストではIVF_PQの6倍以上高速にインデックスを構築できた
- GPU版IVF-RaBitQでは従来手法と比較してストレージも大幅に削減され、Wiki-all（1000万件）データセットでCAGRAの30GB超に対し7.49GBと4分の1以下のストレージで運用可能である

### 2. 主要ベクトルDBでの幅広い採用と実績

- Milvus（v2.6でIVF_RABITQ導入）、Faiss、Elasticsearch/Lucene（BBQとして統合）、LanceDB、VectorChord、CockroachDB、Volcengine OpenSearchなど業界の主要プラットフォームが既に採用済みである
- Elasticsearchでは「BBQ（Better Binary Quantization）」としてプロダクション機能に統合され、PQと比較して構築コストの削減と距離推定速度の向上が実証されている
- TensorChordのVectorChordはPostgreSQL上でRaBitQを活用し、40万ベクトルを1ドル以下のコストで運用するコスト効率の高いベクトル検索を実現した
- NTU VectorDBグループが公式ライブラリ（RaBitQ Library）を公開しており、IVF・HNSW・QGとの組み合わせが参照実装として利用可能。ICML 2025 VecDBワークショップでも発表されている

### 3. 高い検索精度と理論的信頼性

- 理論的に証明された誤差限界O(1/√D)を持つため、次元数が増えるほど精度が向上する予測可能な性能特性を持つ。これにより、512次元以上の高次元エンベディングで特に優れた効果を発揮する
- MilvusのベンチマークではSQ8リファインメント付きで94.7%のリコールを達成し、非圧縮のIVF_FLAT（95.2%）とほぼ同等の精度を維持しながら3.6倍のスループットを実現した
- LanceDBのDBpediaベンチマーク（768次元）ではIVF_PQの約92%に対しRaBitQは96%以上のリコールを達成し、スループットも350QPSから495QPSに向上した
- PQのように経験的にしか性能を保証できない手法と異なり、RaBitQは数学的にデータセット非依存の誤差限界が保証されているため、新しいデータセットでも予測可能な性能を期待できる

### 4. RAG・セマンティック検索・マルチモーダル応用への適合性

- 大規模言語モデル（LLM）のRAG（検索拡張生成）パイプラインでは、高次元エンベディングの大量保存と高速検索が求められるが、RaBitQの極端な圧縮率により同じインフラでより多くのベクトルをメモリ上に保持可能になる
- マルチモーダル検索（テキスト・画像・音声・動画）では各モダリティのエンベディングが高次元になりがちだが、RaBitQは次元数が増えるほど精度が向上するため、高次元マルチモーダルエンベディングとの相性が特に良い
- レコメンデーションシステム、画像検索、セマンティック検索など低レイテンシが求められるリアルタイムアプリケーションでは、ビット演算ベースの高速な距離計算がユーザー体験の向上に直結する
- コードブック不要でデータ分布の変化に再学習なしで対応できるため、継続的にデータが更新されるプロダクション環境での運用負荷が低く、MLOpsの観点からもメンテナンスコストを抑制できる

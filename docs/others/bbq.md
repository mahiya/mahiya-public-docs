# BBQ と DiskBBQ

## 概要

BBQ（Better Binary Quantization）は、Elastic社がApache LuceneおよびElasticsearch向けに開発したベクトル量子化技術である。シンガポール南洋理工大学の研究者が提案した「RaBitQ」から着想を得て、float32ベクトルを1ビット表現にまで圧縮し、約95%のメモリ削減を実現しながら高いランキング品質を維持する。Elasticsearch 8.16でテクニカルプレビューとして導入され、8.18/9.0でコアアルゴリズムが刷新された後、Elasticsearch 9.1では384次元以上のdense_vectorに対するデフォルトの量子化方式に昇格した。

DiskBBQは、BBQの圧縮技術を基盤として構築されたディスクフレンドリーなANN（近似最近傍探索）アルゴリズムである。従来のHNSW（Hierarchical Navigable Small Worlds）がグラフ全体をRAMに保持する必要があるのに対し、DiskBBQは階層的K-meansクラスタリングによりベクトルを小さなクラスタに分割し、クエリ時に関連するクラスタのみをディスクから選択的に読み込む。Elasticsearch 9.2で一般提供が開始され、100MBという極めて少ないメモリ環境でも約15msのクエリレイテンシを維持できることがベンチマークで示されている。

一方、Jina AI社も自社の埋め込みモデル（jina-embeddings-v2/v4/v5）でバイナリ量子化を積極的にサポートしており、Quantization-Aware Training（QAT）により量子化後の精度劣化を最小限に抑える取り組みを行っている。業界全体として、メモリ効率と検索品質を両立するバイナリ量子化技術への注目が高まっている。

## 技術者向けポイント

### 1. BBQの非対称量子化アーキテクチャ

- ドキュメントベクトルは1ビット（バイナリ）に量子化し、クエリベクトルはint4（4ビット、16段階）に量子化する非対称方式を採用している
- セグメント内のベクトル群からセントロイドを増分計算し、各ベクトルをセントロイドからの差分として正規化した上でバイナリ化する
- ドット積計算にはビット演算（AND、ビットカウント、シフト）を使用し、ハードウェアレベルでの高速化を実現している
- Elasticsearch 8.18/9.0では各ベクトルを個別に最適化されたスカラー分位数で量子化する方式に刷新され、リコール精度がさらに向上した

### 2. DiskBBQの階層的K-meansとクラスタベース検索

- ベクトル空間を階層的K-meansで再帰的にパーティション分割し、最大2階層のセントロイドのみを探索してから個別ベクトルをバルクスコアリングする
- `cluster_size`パラメータ（デフォルト384）でクラスタの粒度を制御し、小さいほど精度は上がるがパフォーマンスは低下する
- `visit_percentage`パラメータでセグメント内の探索範囲を制御し、デフォルト0の場合は`k`や`num_candidates`から自動導出される
- GoogleのSOAR（Spilling with Orthogonality-Amplified Residuals）アルゴリズムの変種を用いて、クラスタ境界付近のベクトルを複数クラスタに割り当てることでリコールを改善している

### 3. 低メモリ環境でのパフォーマンス特性

- DiskBBQは合計100MB（JVMヒープ10MB + オフヒープ50MB程度）で1Mベクトルに対し約15msのクエリレイテンシを実現した
- HNSW BBQは同じ100MB環境では動作不能であり、150MBでも289msのレイテンシとなる。メモリ350MB以上でようやくDiskBBQと同等の7ms台に到達する
- インデクシング速度では、DiskBBQはHNSW BBQの約10倍高速（1Mベクトルで約94秒 vs 約1,054秒）であり、メモリ削減時も緩やかに性能が劣化する
- メモリが十分にある環境ではHNSWが2-3ms台の低レイテンシを発揮するが、DiskBBQは800MB/400MBヒープ付近で約2.7msに収束しプラトーに達する

### 4. BBQによるランキング品質の向上メカニズム

- BBQではオーバーサンプリングとfloat32による再ランキングの2段階プロセスにより、圧縮による精度損失を補償している
- Elasticsearch 9.1のデフォルトでは、BBQ使用時の`num_candidates`は`max(1.5*k, oversample*k)`に設定され、float32の`1.5*k`より広い探索範囲を確保する
- BEIRデータセット10種でのNDCG@10ベンチマークにおいて、BBQは10種中9種でfloat32単体のランキング品質を上回った
- OpenSearch FAISSとの比較では、Elasticsearch BBQが最大5倍高速、平均3.9倍のスループットを記録し、Recall@10/50/100の全水準で優位性を示した

## ビジネス側向けポイント

### 1. インフラコストの大幅削減

- BBQはfloat32ベクトルを約32分の1に圧縮し、1億件×384次元のデータで143GBから5.77GBへとメモリ使用量を約95%削減する
- DiskBBQはベクトルインデックス全体をRAMに保持する必要がなく、低コストなディスクベースのインフラで大規模ベクトル検索を運用できる
- Jina AIの試算では、2.5億件の埋め込みに対し、float32で月額約35,021ドルのAWSコストがバイナリ化により月額約1,095ドルに削減される（約97%減）
- 従来HNSWで必要だった大容量RAMの専用ノードが不要となり、汎用的なハードウェアやオブジェクトストレージでの運用が可能になる

### 2. 検索品質を犠牲にしない圧縮

- BBQは圧縮による精度損失をオーバーサンプリングと再ランキングで補償し、float32検索と同等以上のランキング品質を達成している
- Elasticsearch 9.1ではBBQがデフォルト量子化方式となり、特別な設定なしに圧縮とランキング品質の両立が得られる
- Jina AIもQuantization-Aware Trainingにより、バイナリ量子化後のスコア劣化を有意に抑制する学習済みモデルを提供している
- 「高性能モデル（e5-large等）+ BBQ」の組み合わせが、コスト・速度・精度のすべてで「低性能モデル + float32」を上回ることが実証されている

### 3. スケーラビリティとAIアプリケーションの拡張

- DiskBBQはメモリではなくCPUとディスクのみが制約となるため、データ量の増加に対してリニアにスケール可能である
- インデクシング速度がHNSWの10倍高速なため、大量データの初期ロードやリアルタイム更新のユースケースに適している
- RAG（Retrieval Augmented Generation）やセマンティック検索など、ベクトル検索を基盤とするAIアプリケーションの構築コストが大幅に低下する
- Elasticsearch ServerlessおよびElastic Enterpriseで一般提供されており、既存のElasticエコシステムとシームレスに統合できる

### 4. 競合製品に対する優位性

- Elasticsearch BBQはOpenSearch FAISSのバイナリ量子化と比較して、同一リコール水準で最大5倍高速なクエリ速度と平均3.9倍のスループットを達成した
- BBQは8.16の初回リリースから9.0にかけてアルゴリズムが継続的に改善されており、レイテンシとリコールの両面で着実に向上している
- DiskBBQはHNSWの根本的制約（全グラフのRAM常駐）を解消する新しいアプローチであり、競合のベクトルデータベースが抱える同様の課題に対する先行的ソリューションとなっている
- ACORNアルゴリズムとの組み合わせにより、フィルタ付きベクトル検索でも5倍の高速化を実現し、実運用シナリオでの総合的な優位性を確立している

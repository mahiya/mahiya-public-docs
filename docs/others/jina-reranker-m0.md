# jina-reranker-m0

## 概要

jina-reranker-m0 は、Jina AI（現在は Elastic に買収済み）が 2025年4月にリリースした、マルチモーダル・多言語対応のリランカーモデルである。24億パラメータ（Qwen2-VL-2B ベース）のデコーダーオンリー型ビジョン言語モデルとして構築されており、テキストクエリに対してテキスト文書と画像文書の両方を統一的にランキングできる点が最大の特徴である。従来のクロスエンコーダー型リランカーとは異なり、Vision Transformer（ViT）と大規模言語モデル（LLM）を LoRA でファインチューニングし、MLP でランキングロジットを生成するアーキテクチャを採用している。29言語に対応し、PDF、スキャン画像、スクリーンショット、図表を含むビジュアルリッチな文書の検索精度向上に威力を発揮する。Jina AI の Reranker API、HuggingFace、AWS/Azure/GCP で利用可能であり、Elastic Inference Service（EIS）への統合も予定されている。ライセンスは CC-BY-NC-4.0（非商用利用は無料、商用利用は API 経由）。

## 技術者向けポイント

### 1. デコーダーオンリー VLM アーキテクチャによるモダリティギャップの解消

- Qwen2-VL-2B をベースとし、ViT + プロジェクタ + LoRA ファインチューニング済み LLM + MLP という構成で、クエリと文書の関連性スコアを生成する識別モデルとして動作する
- 従来のエンコーダーオンリーモデル（jina-clip-v1/v2 等）では、テキストとテキスト、画像と画像がそれぞれクラスタリングされる「モダリティギャップ」が問題となっていたが、デコーダーオンリー型にすることでこのギャップをほぼ解消している
- 画像入力時は ViT とプロジェクタが隣接する 2x2 トークンを 1 つのビジュアルトークンに圧縮し、特殊トークン `<|vision_start|>` / `<|vision_end|>` でビジュアルトークンの境界を明示する
- 最大 32K トークン、画像は 56x56 ピクセルから 4K 解像度まで対応し、大きな画像は 28x28 ピクセルのパッチが 768 個以下になるようリサイズされる

### 2. ベンチマーク性能：テキスト・多言語・コード・ビジュアル全方位で高スコア

- テキスト間リランキング（BEIR）で NDCG@10 = 58.95 を達成し、jina-embeddings-v3（55.81）や bge-reranker-v2-m3（56.51）を上回る
- 多言語ベンチマーク（MIRACL、18言語）で NDCG@10 = 66.75、長文書ベンチマーク（MLDR、13言語）で NDCG@10 = 59.83 を記録
- コード検索（CoIR）で NDCG@10 = 63.55 と競合モデルを大幅に上回り、ビジュアル文書検索（ViDoRe）では NDCG@5 = 91.02 という突出した性能を示す
- 視覚・言語の構成的推論ベンチマーク（Winoground）でも平均スコア 43.92 を達成し、画像とテキストの複合的な理解力を実証している

### 3. ペアワイズリランキングと 2段階パイプラインの設計

- jina-reranker-m0 はペアワイズ方式で動作し、1 つのクエリに対して 2 つの候補文書（テキスト or 画像）のどちらがより関連性が高いかを判定する
- リストワイズ方式が必要な場合は jina-reranker-v3（6億パラメータ、テキストのみ、最大 64 文書、131K トークン）と使い分ける設計になっている
- 実用的なパイプラインとしては、jina-clip-v2 で初期候補を取得し、jina-reranker-m0 で 2 段階目のリランキングを行う構成が推奨されており、テキストと画像の両モダリティからの候補を混合してリランクすると Recall@10 が最も向上する（EDIS データセットで 36.24）
- 入力上限はクエリテキスト＋候補文書を合わせて 10,240 トークンで、画像の 28x28 ピクセルタイル 1 枚が 1 トークンとしてカウントされる

### 4. Elastic Inference Service（EIS）との統合と実装

- Jina AI が Elastic に買収されたことで、Elasticsearch のネイティブ推論サービスとして Jina モデル群が利用可能になっている（現時点では v2/v3 が EIS 上で稼働中、m0 は次期対応予定）
- BM25F によるレキシカル検索 + jina-embeddings-v3 によるベクトル検索 + Jina リランカーによるリランキングという、ハイブリッド検索パイプラインを Elasticsearch 内で完結できる
- HuggingFace からローカルで利用する場合は、API にはない画像クエリの使用やマルチモーダル文書の直接入力など、より柔軟な運用が可能である
- Google Colab ノートブックが公式に提供されており、ローカル実行のデモと実装パターンを確認できる

## ビジネス側向けポイント

### 1. PDF・スキャン画像・スクリーンショットなどビジュアル文書の検索精度を大幅に改善

- 企業内に蓄積された PDF、スキャンした契約書、スクリーンショット、スライド資料など、テキスト抽出が困難なビジュアルリッチ文書に対して、画像のまま検索・ランキングが可能
- 従来はOCRでテキスト変換してから検索する必要があったが、jina-reranker-m0 は画像を直接理解して関連性を判定するため、OCR エラーや書式崩れによる検索精度の低下を回避できる
- ビジュアル文書検索のベンチマーク（ViDoRe）で NDCG@5 = 91.02 という業界最高水準を達成しており、図表・インフォグラフィック・複雑なレイアウトの文書にも対応する
- 29言語対応のため、多言語環境のグローバル企業でも単一モデルで横断的な文書検索が実現できる

### 2. 既存検索システムへのドロップイン統合が可能

- リランカーは既存の検索パイプラインの後段に追加するだけで精度を向上させるため、インデックスの再構築やシステム全体の変更なしに AI 検索のメリットを享受できる
- Elasticsearch ユーザーは EIS 経由で API を呼び出すだけで利用でき、GPU のセットアップやホスティングの複雑さは不要
- BM25 などのキーワード検索やハイブリッド検索の結果に対して後段で適用でき、RAG（検索拡張生成）ワークフローの精度向上にも直結する
- Jina AI の API 経由での利用に加え、HuggingFace からのセルフホスティング、AWS/Azure/GCP のマネージド環境での利用も可能

### 3. オープンソースで非商用無料、商用は API ベースの柔軟なライセンス

- CC-BY-NC-4.0 ライセンスにより、研究・学術・社内評価などの非商用用途では無料で利用可能
- 商用利用は Jina AI（Elastic）の API を通じて従量課金で利用でき、初期投資なしでスモールスタートが可能
- HuggingFace 上でモデルの重みが公開されているため、技術チームが事前に性能を検証してから導入判断ができる
- Elastic への買収により、Elasticsearch エコシステムの一部として長期的なサポートと継続的な改善が見込まれる

### 4. jina-reranker-v3 との使い分けでコストと精度を最適化

- テキストのみの検索用途には jina-reranker-v3（6億パラメータ、リストワイズ、131K トークン対応）がより軽量・高効率で、一度に最大 64 文書をリランキングできる
- 画像・PDF・混合コンテンツを含む用途には jina-reranker-m0（24億パラメータ、ペアワイズ）が必要となるが、モデルサイズが大きい分、推論コストが高くなる
- 2 つのモデルを用途に応じて使い分けることで、検索精度とインフラコストのバランスを最適化できる
- いずれのモデルも同じ API エンドポイント体系で利用でき、切り替えコストが低い

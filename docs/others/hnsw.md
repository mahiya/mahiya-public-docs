# HNSW (Hierarchical Navigable Small World)

## 概要

HNSW (Hierarchical Navigable Small World) は、高次元ベクトル空間における近似最近傍探索 (ANN: Approximate Nearest Neighbor) のためのグラフベースのアルゴリズムである。ベクトルをノード、ベクトル間の近接関係をエッジとする多層グラフ構造を構築し、上位層から下位層へ段階的に探索を絞り込むことで、全件走査 (brute-force) と比較して対数的な計算量で高精度な近傍探索を実現する。最上位層には少数のノードが「高速道路」のように配置され、下位層に進むにつれノード数が増加し、最下位層には全ノードが含まれる。探索時はエントリポイントから最上位層で大まかな近傍を特定し、各層で最も近いノードを次層のエントリポイントとして降下していく。Elasticsearch / Apache Lucene をはじめ、多くのベクトル検索エンジンで採用されている業界標準のアルゴリズムである。

## 技術者向けポイント

### 1. グラフ構築パラメータとトレードオフ

- `M` はグラフの各ノードが持つ最大エッジ数を制御し、値を大きくするとリコールが向上するがインデックスサイズとメモリ使用量が増加する
- `ef_construction` はインデックス構築時の探索幅を決定し、大きいほど高品質なグラフが構築されるが構築時間が長くなる
- `num_candidates` (= `ef_search`) は検索時の候補ノード数であり、リコールとレイテンシのバランスを直接制御するパラメータである
- これらのパラメータはデータ分布・次元数・要求精度に応じてチューニングが必要であり、一律の最適値は存在しない

### 2. 早期終了 (Early Termination) による探索の最適化

- HNSW のグラフ探索では、探索が進むにつれて新たな真の近傍が見つかる確率が収束し、冗長な距離計算が発生する
- 固定的な早期終了（訪問ノード数上限・タイムアウト）は実装が容易だが、クエリごとの難易度差に対応できない
- Elasticsearch 9.3 で導入された適応的早期終了は、Discovery Rate の移動平均と標準偏差を Welford のアルゴリズムで追跡し、クエリごとに動的に打ち切り判定を行う
- これにより、容易なクエリでは過剰探索を削減し、困難なクエリでは十分な探索を維持することで、リコールを保ちながらレイテンシを改善できる

### 3. フィルタ付きベクトル検索の課題と対策

- HNSW グラフはフィルタ条件を考慮せずに構築されるため、フィルタ適用時にグラフの接続性が断たれ、探索が局所最適に陥りやすい
- フィルタ対象外のノードもグラフ探索の経由には必要であり、直感に反してフィルタで候補を減らすほど距離計算回数が増加するケースがある
- ACORN アルゴリズムは、フィルタ済みノード間の到達可能性を維持しながら効率的にグラフを探索する手法として Lucene に導入された
- Pre-filtering と Post-filtering の使い分けはフィルタの選択率に依存し、選択率が低い場合は線形スキャン、高い場合はグラフ探索が有利となる

### 4. セグメントベースのグラフ管理とマージ戦略

- Lucene では各セグメントが独立した HNSW グラフを持ち、検索時は複数セグメントを個別に探索して結果をマージする
- セグメントマージ時に従来は最大セグメントのグラフを保持し他のベクトルを再挿入していたが、既存グラフの近接情報が破棄されていた
- 新しいマージ戦略では Join Set を用いて既存グラフの近傍情報を活用し、再挿入時の距離計算を大幅に削減する
- DiskBBQ は HNSW の代替として階層的 K-means + BBQ 量子化によりディスクベースの探索を実現し、RAM 制約下でのスケーラビリティを改善する

## ビジネス側向けポイント

### 1. ベクトル検索の高速化によるユーザー体験の向上

- HNSW により、数百万〜数十億件のデータに対してミリ秒単位で類似検索が可能となり、リアルタイムな検索体験を提供できる
- 全件走査と比較してデータ量の増加に対し対数的にしかコストが増えないため、データ規模の拡大に伴う性能劣化が緩やかである
- 商品レコメンデーション、画像検索、セマンティック検索など、AI を活用した検索機能の基盤技術として広く採用されている
- Elasticsearch 8.0 以降で標準サポートされており、既存の Elasticsearch 基盤を活かしたベクトル検索の導入が可能である

### 2. 精度と速度のバランスを業務要件に合わせて調整可能

- パラメータ調整により「少し精度を下げてでも高速に返す」「時間をかけてでも高精度な結果を返す」といった要件に柔軟に対応できる
- 適応的早期終了により、簡単なクエリは素早く返し、難しいクエリには十分な計算リソースを割り当てる自動最適化が可能になった
- これにより、検索 SLA（応答時間目標）を守りながら、検索品質を最大化するという両立が現実的になっている
- ユースケースに応じたチューニングにより、同一ハードウェアでもスループット（単位時間あたりの処理件数）を大幅に改善できる

### 3. インフラコストの最適化とスケーラビリティ

- HNSW は高速だが全ベクトルを RAM 上に保持する必要があり、大規模データではメモリコストが課題となる
- 量子化技術（BBQ 等）との組み合わせにより、ベクトルのメモリフットプリントを大幅に削減しつつ精度を維持できる
- DiskBBQ のようなディスクベースの代替手法により、RAM に収まらない規模のデータでも実用的な検索性能を確保できる
- クラウド環境ではメモリ最適化により、より小さいインスタンスタイプの選択が可能となり、直接的なコスト削減につながる

### 4. フィルタリングとの組み合わせによる実用的な検索の実現

- 実際のビジネスでは「カテゴリ」「価格帯」「在庫状況」などの条件付きで類似検索を行う要件が大半であり、フィルタ対応は必須である
- Elasticsearch ではフィルタ付きベクトル検索が標準機能として提供されており、従来の全文検索との組み合わせ（ハイブリッド検索）も可能である
- 最新の ACORN アルゴリズム採用により、フィルタ条件が厳しい場合でも検索品質の劣化を抑えた高速検索が可能になっている
- 既存のビジネスロジック（権限制御・地域制限・属性フィルタ等）をベクトル検索と統合でき、実運用への適用障壁が低い
